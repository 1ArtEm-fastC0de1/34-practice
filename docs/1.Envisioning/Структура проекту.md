## **1. ЦІЛІ ТА ЗАВДАННЯ**

**Мета 1: Реалізувати базовий функціонал нотатника**

- створення нової нотатки;

- редагування існуючої нотатки;

- перегляд нотатки;

- видалення нотатки;

- автозбереження під час редагування.

**Мета 2: Забезпечити підтримку Markdown**

- інтеграція Markdown-парсера;

- відображення форматованого тексту;

- перемикання між режимами редагування та перегляду кнопкою (іконка олівця).

**Мета 3: Реалізувати експорт**

- експорт однієї нотатки у форматі .md;

- експорт усіх нотаток;

- коректне формування файлів для завантаження.

**Мета 4: Забезпечити зручний інтерфейс**

- адаптивна верстка (mobile-first);

- світла та темна тема;

- простий мінімалістичний інтерфейс;

- автоматичне збереження змін.

## **2. ПРИПУЩЕННЯ І ОБМЕЖЕННЯ**

**Припущення**

- користувач працює в сучасному браузері (Chrome, Edge, Firefox);

- збереження даних здійснюється через LocalStorage;

- автозбереження відбувається автоматично при зміні тексту;

- імпорт нотаток може бути реалізований як додаткова функція;

- достатньо двох тем: світла і темна;

- застосунок не потребує серверної частини.

**Обмеження**

- термін реалізації - 1 місяць;

- проект реалізується без бекенду;

- відсутня система авторизації;

- drag-and-drop не реалізується;

- налаштування шрифтів не входить до поточного релізу;

- експорт лише у форматі .md.

## **3. РАМКИ ПРОЕКТУ**

**Функціональність рішення**

- CRUD-операції для нотаток;

- автозбереження;

- підтримка Markdown;

- перемикання режиму редагування/перегляду;

- експорт однієї та всіх нотаток;

- світла/темна тема;

- адаптивний дизайн.

**За рамками рішення**

- теги та пошук;

- сортування/фільтрація;

- функція «закріпити» нотатку;

- імпорт (можлива майбутня версія);

- хмарна синхронізація;

- авторизація;

- експорт у PDF/TXT.

### **3.3. Кошториси проекту**

### **3.4. План-графік проекту**

На цьому етапі будується перший варіант *графіка виконання (календарного графіка) проекту* на основі виділених віх і вже сформульованих завдань, для кожної з яких задаються дати початку і закінчення.

Процес побудови графіка проекту є *ітеративним*. На фазі створення концепції (аналізу) графік будується для основних віх проекту. На фазі планування графік стає все детальнішим в процесі виділення окремих завдань проекту.

:point_right: На основі виділених вище віх, а також сформульованих цілей і завдань визначте очікувану часову протяжність проекту і побудуйте попередній графік виконання проекту.

## **5. ПРОТОКОЛИ ПРОЕКТУ**

### **5.1. Керування ризиками**

**Методи керування ризиками:**

- ідентифікація ризиків на початку кожного тижня

- ведення таблиці ризиків

- оцінка ймовірності та впливу

- планування заходів реагування

**Основні ризики:**

| Ризик                      | Ймовірність | Вплив    | Стратегія                  |
| -------------------------- | ----------- | -------- | -------------------------- |
| Недостатній час            | Середня     | Високий  | Пріоритезація функцій      |
| Технічні труднощі з React  | Низька      | Середній | Консультації, документація |
| Проблеми з автозбереженням | Середня     | Середній | Додаткове тестування       |

**Відповідальні:**

- Менеджер програми - контроль ризиків

- Розробники - технічні ризики

- Тестувальники - перевірка стабільності

### **5.2. Керування конфігурацією**

У проекті використовується система контролю версій Git та платформа GitHub.

**Правила роботи:**

- основна гілка - main

- розробка - зберігається у репозиторії в папці /src

- внесення змін - через Pull Request

- код-рев’ю перед злиттям

Документація зберігається у репозиторії в папці /docs.

### **5.3. Керування змінами**

**Процес внесення змін:**

1. Створення запиту на зміну (Issue).

2. Обговорення командою.

3. Аналіз впливу на трикутник компромісів.

4. Прийняття або відхилення зміни.

5. Реалізація та тестування.

Якщо зміна впливає на терміни або функціональність — погоджується із замовником.

### **5.4. Керування впровадженнями**

Застосунок буде розгорнуто як SPA на платформі GitHub Pages.

**Процес релізу:**

- створення production-збірки

- перевірка працездатності

- публікація через GitHub Pages

- створення Release на GitHub

Оновлення виконуються шляхом повторного деплою.
